01Feature Selection
Interview tip: Notification systems look simple but hide real complexity in deduplication, user preferences, and multi-channel fan-out. Anchor the design on: what happens when a service retries? The answer (idempotency) should drive everything.
FEATURE 01
Multi-Channel Dispatch
Fan-out a single notification request to SMS (Twilio), push notifications (APNS/FCM), and email (SES). One intent, multiple delivery channels.
FEATURE 02
User Preferences & Rate Limiting
Respect per-user opt-ins, quiet hours, and frequency caps per channel. Suppress or delay based on user settings — never spam.
FEATURE 03
Guaranteed Delivery with Retry
Transactional notifications (order confirm) must be delivered. DLQ with exponential backoff for failed providers. Deduplication prevents duplicates on retry.
⬇ Below the Line (Out of Scope)
Read receipts & delivery confirmation UI  ·  ML-based send-time optimization  ·  A/B testing notification copy  ·  Two-way SMS replies  ·  Notification center in-app UI
Why mention it: Each is a product feature on top of the delivery infrastructure. Naming them shows you understand the full product without letting them derail the architecture.
02Non-Functional Requirements
Interview tip: Not all notifications are equal. Map NFRs to priority tiers: TRANSACTIONAL (order confirm) requires at-least-once delivery; PROMOTIONAL is best-effort, no retry. Mixing them is a common mistake.

User opts out → no notifications sent within 60s. Redis cache write-through. TCPA compliance requires immediate effect for SMS.
Preferences

Guaranteed Delivery

Order confirmations must arrive. Max 4 retry attempts with exponential backoff. After 4 failures → DLQ + ops alert.
System-Wide

A retry must never result in duplicate notifications. Deduplication via Redis idempotency key per (request, channel). This is the hardest invariant to maintain.

04Core Entities
Interview tip: The key distinction: NotificationRequest is the intent (one per event); NotificationDelivery is the execution record (one per channel per request). This 1→N fan-out is the core data model.
📬 NotificationRequest

The incoming intent. One per event from any producer service. Carries the idempotency key that prevents duplicate dispatches.

request_id · user_id · event_type · template_id · template_vars · idempotency_key · priority
👤 UserPreference

Per-user channel settings. Opt-ins, push token, quiet hours, and frequency caps. Cached in Redis for sub-ms access on every dispatch.

user_id · sms_opted_in · email_opted_in · push_token · quiet_start · quiet_end · timezone · freq_caps{}
📋 NotificationDelivery

Delivery attempt record per channel. One NotificationRequest fans out to N NotificationDelivery rows (one per channel). Tracks retry state.

delivery_id · request_id · channel · status · provider_msg_id · attempt_count · next_retry_at · created_at
📝 NotificationTemplate

Reusable template with variable slots. Versioned. The Template Service renders personalized content before dispatch.

template_id · event_type · channel · subject_template · body_template · cta_url · version
❌ DeadLetterEvent

A notification that exhausted all retries. Stored for ops review and manual requeue. Triggers an alert to the on-call team.

dl_id · delivery_id · last_error · final_attempt_at · channel · user_id · event_type
📊 RateLimitBucket

Redis counter for sliding window rate limiting. One key per (user, channel, time window). Atomic INCR prevents exceeding frequency caps.

Key: rl:{user_id}:{channel}:{epoch_hour} | Value: count | TTL: 2h
Key relationship: NotificationRequest is the intent; NotificationDelivery is the execution per channel. One request → 3 deliveries (SMS + push + email). Like one invoice generating 3 separate courier shipments — same content, different carriers.

Idempotency is the hardest invariant: Every POST /v1/notifications requires an idempotency_key. If a checkout service retries on timeout with the same key, the Notification Service must return the original request_id without re-dispatching. One order confirm → one notification, no matter how many retries.

07Data Model
Interview tip: Four distinct storage concerns. Redis handles the latency-sensitive operations (preferences, dedup, rate limiting). Postgres handles the durable delivery record.

Redis — Three separate concerns
Key PatternTypePurpose & TTL
prefs:{user_id}HashUser preferences cache. Fields: sms_opted_in, push_token, quiet_start, quiet_end, tz. TTL: 60s write-through.
dedup:{idem_key}:{channel}StringDeduplication cache. Value: delivery_id. TTL: 24h. SET NX to atomically check-and-set.
rl:{user_id}:{channel}:{epoch_hr}String (INCR)Rate limit sliding window counter. TTL: 2h. Atomic INCR — no race conditions.
Rate limiting in Redis with INCR is O(1) and atomic. Never implement rate limiting in your application DB — at 1,150 notifications/sec that's 1,150 read-then-write transactions per second on your delivery DB. Redis handles this in microseconds with zero contention.
08Tradeoffs & Deep Dives
Interview tip: Use the BAD → GOOD → GREAT pattern. For notifications, the three deepest problems are: deduplication (hardest to get right), quiet hours (surprising complexity), and provider outages (multi-provider failover).
NFR: ZERO DUPLICATE NOTIFICATIONS

How do you prevent sending 3 order confirmations when the checkout service retries on timeout?

❌ BAD — No idempotency

Checkout service times out → retries POST /v1/notifications 3 times → user gets 3 order confirmation SMS. Customer calls support. Story goes viral on Twitter. On Black Friday this becomes a scale incident — millions of duplicates. Regulators flag TCPA violations for the SMS duplicates.

⚠ GOOD — DB unique constraint on (request_id, channel)

Add UNIQUE(request_id, channel) to notification_deliveries. Duplicate INSERT throws constraint violation → return original delivery_id. Works but: DB write on every dispatch attempt at 1,150/s adds lock contention; doesn't cleanly return the original response body; slow under high retry load during peak.

✅ GREAT — Two-layer deduplication: Redis fast path + DB safety net

Layer 1: Before dispatching each channel, SET NX dedup:{idem_key}:{channel} = delivery_id EX 86400. NX = only set if not exists. If key already set → skip dispatch (already sent), return original delivery_id. O(1), <1ms, atomic. Layer 2: DB UNIQUE constraint as safety net if Redis is cold after restart. Redis is the fast path (catches 99.9% of duplicates in microseconds). DB constraint catches the rare Redis miss. Net result: zero duplicate notifications delivered to users, <1ms overhead per dispatch attempt.

NFR: TRANSACTIONAL DELIVERY DURING QUIET HOURS

A user has quiet hours 11pm–8am and places an order at midnight. How do they still get their order confirmation?

❌ BAD — Drop the notification silently

User places order at midnight, has quiet hours set for SMS. Notification service drops the order confirmation. User wakes up at 8am with no confirmation, thinks order failed, calls support or reorders. TRANSACTIONAL notifications should never be silently dropped.

⚠ GOOD — Cron job scans for suppressed notifications

Store suppressed notifications with a send_after timestamp. Cron job runs every 5 minutes, queries for rows where send_after < NOW() and status=SUPPRESSED. Works but: polling DB at 1,150 notifications/s scale is expensive; 5-minute resolution means messages send up to 5 min late; cron becomes a bottleneck at scale.

✅ GREAT — Delayed Kafka messages for transactional, drop for promotional

When dispatcher detects quiet hours: if TRANSACTIONAL → re-publish to Kafka with message timestamp = quiet_end_time_utc. Kafka holds the message until that timestamp and delivers it to the consumer at the right time. No polling, no DB scan, no application state. If PROMOTIONAL → drop entirely (no re-delivery — user opted for quiet hours, promotional content has a freshness TTL). Transactional messages guaranteed to arrive at the first minute after quiet hours end. At-exactly-one-delivery-window semantics.

NFR: DELIVERY SLA DURING PROVIDER OUTAGE

Twilio goes down for 20 minutes during Black Friday. How do transactional SMS notifications still get delivered?

❌ BAD — Single SMS provider (Twilio only)

Twilio outage = zero SMS delivery for 20 minutes. 20 min × 560 SMS/s = 672,000 undelivered order confirmations. All go to DLQ. Manual retry after outage resolves — delayed delivery to users. SLA breach.

⚠ GOOD — Retry against same provider with exponential backoff

On Twilio failure: retry at +30s, +5min, +30min. During a 20-minute outage, all retries also fail. After 4 attempts → DLQ. User gets no SMS. Backoff helps with transient errors but not sustained outages. The retries also exhaust Twilio's rate limit when it comes back online (retry storm).

✅ GREAT — Multi-provider circuit breaker with automatic failover

Primary SMS: Twilio. Fallback: AWS SNS. Circuit breaker per provider (state stored in Redis, shared across all worker pods): if error rate >5% in a 60-second sliding window → open circuit, route all SMS to fallback provider for 5 minutes, then probe with a single request. If probe succeeds → close circuit and return to primary. During Twilio outage: circuit opens within 60s, all traffic shifts to SNS automatically. For transactional notifications that fail both providers: escalate to email as last resort (delivery channel downgrade, logged for ops). No user sees zero delivery unless both SMS providers and email all fail simultaneously.

Walmart at Scale: Walmart sends 500M+ notifications per month across 100M+ customers — order confirmations, shipping updates, price drop alerts, and promotional offers. On Black Friday midnight opening, they send 10M+ order confirmation notifications in a 15-minute window — a 40× normal burst. Their platform pre-warms Twilio and SES connection pools 30 minutes before peak, uses Kafka consumer group auto-scaling to add dispatch workers dynamically, and enforces a promotional notification blackout window during the first 2 hours of Black Friday to reserve all channel capacity for transactional order-confirmation traffic.
