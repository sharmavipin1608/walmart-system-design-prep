01 Feature Selection
Interview tip: Don't just list features — present 5-6 candidates and collaboratively narrow to 3 with the interviewer. This signals product thinking and keeps them engaged.
FEATURE 01
Real-Time Stock Tracking
Track inventory levels across all stores and warehouses in near real-time. Every sale, return, or restock updates the count.
FEATURE 02
Inventory Reservation
Reserve stock during checkout to prevent overselling. Reservations expire if payment doesn't complete within a TTL window.
FEATURE 03
Low Stock Alerting & Restocking
Trigger alerts and auto-generate purchase orders when stock falls below configurable thresholds per SKU per location.
⬇ Below the Line (Out of Scope)
Supplier management & vendor onboarding  ·  Demand forecasting & ML-based replenishment  ·  Dynamic pricing engine  ·  Returns fraud detection  ·  Multi-currency / international tax handling

02 Non-Functional Requirements
Interview tip: Map NFRs to each feature. Generic NFRs (just listing "scalability, availability") is a junior signal.
Stock Tracking
Eventual Consistency
Read counts can lag by ~1–2s. Analytics dashboards don't need perfect freshness.
Reservation
Strong Consistency
No overselling. Reservation must be atomic — use optimistic locking or Redis atomic ops.
Alerting
At-Least-Once Delivery
Missing a low-stock alert is worse than a duplicate. Make consumers idempotent.
System-Wide
High Availability (99.99%)
~52 min downtime/year. Multi-AZ deployments, no single points of failure.
System-Wide
Peak Load: Black Friday
10x normal traffic. Auto-scaling, circuit breakers, and graceful degradation required.
Stock Tracking
Low Write Latency
Stock updates must be persisted within 500ms of a sale event to prevent stale reads at checkout.

Monitoring & alerting infrastructure  ·  GDPR / PII compliance  ·  Multi-region active-active replication  ·  Audit logging for regulatory requirements

04 Core Entities
Interview tip: Define entities before API — they anchor your data model and make your API shapes obvious. Keep it to 3–5 entities. Don't list every field yet.
📦 SKU

A product variant (size, color, etc.). Has a global identity but exists at many locations with different stock levels. The unit of inventory tracking.

sku_id · name · category · unit_of_measure
🏪 Location

A store or warehouse. Each location has its own stock levels per SKU. Drives sharding strategy — most queries are location-scoped.

location_id · type (STORE|WAREHOUSE) · region
📊 InventoryRecord

The current stock state for a (SKU, Location) pair. This is the hot read/write entity — it must be strongly consistent for reservations and efficiently cached for reads.

sku_id · location_id · available_qty · reserved_qty · version
📋 InventoryEvent

An immutable log of every stock change (sale, return, restock, damage). Source of truth for audit and analytics. Drives the Kafka event stream.

event_id · sku_id · location_id · delta · reason · occurred_at
🔒 Reservation

A short-lived hold on stock during checkout. Has a TTL — auto-expires if payment doesn't complete. Prevents overselling without permanently reducing available_qty until payment confirms.

reservation_id · sku_id · location_id · qty · order_id · expires_at
🔔 AlertThreshold

Configuration for when to fire a low-stock alert. Stored per (SKU, Location). Evaluated by the Alert Service on every inventory event.

sku_id · location_id · threshold_qty · alert_channel
Key relationship: InventoryRecord is the live state. InventoryEvent is the append-only history. Think of it like a bank account balance (record) vs transaction ledger (events).

08 Tradeoffs & Deep Dives
Interview tip: Each deep dive is triggered by an NFR. Lead with the bad solution, explain why it fails, then escalate to good and great. This shows systematic thinking — not just pattern matching.
NFR: Strong Consistency for Reservations
How do we prevent overselling under concurrent checkouts?
❌ BAD — DB-level SELECT then UPDATE
Read stock qty, check if sufficient, then write deduction in two separate DB calls. Race condition: two concurrent checkouts both read qty=1 and both succeed, resulting in qty=-1.
⚠ GOOD — Optimistic Locking in Postgres
Add a version column. UPDATE … WHERE version=N fails if another write happened concurrently. Retry on failure. Works but adds DB round-trips and retry logic under high contention (Black Friday).
✅ GREAT — Redis Atomic DECR with floor check
Use a Lua script on Redis: atomically check available_qty > 0 and decrement in a single operation. Sub-millisecond, no race conditions. Postgres is updated asynchronously via Kafka. Redis AOF persistence handles crash recovery.
NFR: High Availability under Peak Load (Black Friday)
How do we handle 10x traffic without the DB becoming a bottleneck?
❌ BAD — Scale DB vertically
Throw bigger hardware at Postgres. Hits physical limits quickly, expensive, single point of failure, and doesn't solve the cold cache problem on startup.
⚠ GOOD — Add Read Replicas + Shard Postgres
Shard by location_id, add read replicas per shard. Reduces per-shard load. Still risks cold cache stampede and hot shards for high-volume stores (supercenters).
✅ GREAT — Pre-warm Redis + Circuit Breaker + TTL tightening
Three layers:
1. Pre-warm Redis the night before with all hot SKU counts — prevents cold cache stampede at midnight.
2. Circuit breaker on Postgres writes — if DB is overwhelmed, buffer in Kafka and drain after peak. Serve reads from Redis only.
3. TTL tightening — reduce reservation TTL from 300s → 60s to recycle abandoned carts faster and maximize available stock.
NFR: At-Least-Once Delivery for Alerts + Idempotency
How do we handle POS retries without double-deducting inventory?
❌ BAD — No idempotency, trust the network
POS terminals retry on timeout. Without deduplication, a retry fires a second stock deduction. Stock goes negative, customer order succeeds but item is unavailable.
⚠ GOOD — DB-level dedup table
Store processed transaction_ids in a DB table with a UNIQUE constraint. On retry, INSERT fails, skip processing. Works but adds a DB round-trip on every write — bottleneck at scale.
✅ GREAT — Redis SET dedup with 24hr TTL
Store idempotency_key in a Redis SET with 24hr TTL. SETNX check is sub-millisecond and doesn't touch Postgres. POS retries within 24hrs are safely deduplicated. Combine with Kafka exactly-once semantics (transactional producer) for the event pipeline.
Walmart-Specific: Offline Store Behavior
Stores must operate during network outages. POS terminals buffer events locally (SQLite or embedded DB) and sync to central system on reconnect. The idempotency_key + event deduplication handles replay safely. This is a strong signal to mention — it shows awareness of Walmart's physical store constraints vs pure cloud-native thinking.
