Keyword search over 500M products with category, price range, brand, and rating filters. Return ranked results with facet counts.
FEATURE 02

Prefix-based suggestions as user types. Must respond in <50ms. Powered by precomputed popularity scores, not live ES queries.

02Non-Functional Requirements
Interview tip: Search is read-heavy (100:1). Every NFR should reflect that — cache aggressively, replicate search nodes, degrade gracefully (stale results > no results).
Full-Text Search
Low Latency: p99 < 200ms
Search latency directly impacts conversion. Cache popular queries. ES must return in <100ms to leave budget for network + rendering.
Full-Text Search
High Availability: 99.9%
Search down = revenue down. 3-replica ES shards, CDN for popular result caching, circuit breaker to return cached results on ES failure.
Typeahead
Ultra-Low Latency: p99 < 50ms
User is typing — every 100ms delay is felt. Redis sorted set prefix lookup, never query Elasticsearch on every keypress.
Index Updates
Freshness: < 30s Lag
Price changes during flash sales must appear quickly. CDC pipeline target: 5–10s end-to-end from DB commit to ES indexed.
System-Wide
Read-Heavy: 100:1 R/W
500K QPS peak read, 5K QPS index updates. Optimize entirely for reads. Elasticsearch replicas absorb read load, not writes.
System-Wide
Graceful Degradation
ES cluster overloaded → serve cached results. Cache empty → return top-selling items. Never show empty search results page.
⬇ Below the Line (Out of Scope)
ML re-ranking per user  ·  Multi-language & locale-specific search  ·  Catalog deduplication  ·  GDPR right-to-erasure from search index

04Core Entities
Interview tip: Distinguish the source-of-truth entity (Product in catalog DB) from the search-optimized read model (ProductIndex in ES). This CQRS distinction is a senior signal.
📦 Product

Source of truth in the product catalog DB. Has all fields. Changes here trigger index update events via CDC.

product_id · title · description · brand · category_path · price · seller_id · updated_at
🔍 ProductIndex

Denormalized Elasticsearch document optimized for search. Includes precomputed signals like in_stock and popularity_score. This is the read model.

product_id · title · brand · category · price · avg_rating · in_stock · popularity_score · last_indexed_at
🔎 SearchQuery

Incoming query with filters and pagination. The object the Search Service processes on every request.

query_text · brand[] · price_min · price_max · rating_min · category · page · page_size
📊 SearchResult

The returned result set. Includes facet counts for UI filters. Cached in Redis keyed by the query fingerprint.

query_id · hits[{product_id,title,price,score}] · total · facets · latency_ms
💡 TypeaheadSuggestion

Precomputed prefix → suggestion mapping stored in Redis sorted sets. Score = popularity. Refreshed nightly from search logs.

prefix · suggestions[{text, product_count, score}]
📝 IndexUpdateEvent

Kafka event triggering a re-index. Carries only the changed fields — no need to re-index the full document on a price change.

event_type · product_id · changed_fields{} · occurred_at
Key relationship: Product is the write model; ProductIndex is the read model. They're kept in sync via IndexUpdateEvent — like a CQRS pattern where ES is the read store and Postgres is the write store.

08Tradeoffs & Deep Dives
Interview tip: Use the BAD → GOOD → GREAT pattern for each NFR. The GREAT answer for search is almost always: separate the query path from the indexing path, and cache aggressively at every layer.
NFR: RELEVANCE RANKING

How do you make search results feel relevant, not just keyword-matched?

❌ BAD — Pure TF-IDF

Elasticsearch default BM25. "iPhone case" returns any document containing those words, ranked by keyword frequency. Obscure products with exact keyword match outrank popular OtterBox cases. No business context. Kills conversion.

⚠ GOOD — Static boosts

function_score query with static multipliers: avg_rating × 1.2, in_stock × 1.5, brand_tier × 1.1. Better but static — can't learn from what users actually click and buy. Seasonal trends (Black Friday) not captured.

✅ GREAT — Two-phase ranking with behavioral signals

Phase 1: Elasticsearch BM25 retrieves top-100 candidates (<50ms). Phase 2: Lightweight gradient boosting model re-ranks top-100 using precomputed signals: click-through rate (CTR), add-to-cart rate, purchase rate, avg_rating, review count. Signals computed nightly from clickstream data and stored as popularity_score in the ES document. Online re-ranking: <15ms on 100 candidates, not 500M. Total latency stays under 200ms. Model retrained weekly on fresh behavioral data.

NFR: TYPEAHEAD AT 290K QPS

How do you serve prefix suggestions in <50ms at 290K QPS peak?

❌ BAD — Query Elasticsearch on every keypress

Wildcard prefix query on ES. "iph*" scans posting lists. 200ms+ latency. At 290K QPS this obliterates the ES cluster that's already serving search results. ES not designed for this access pattern.

⚠ GOOD — Database prefix query

MySQL/Postgres full-text index on suggestion strings. SELECT text WHERE text LIKE 'iph%'. 10–20ms with B-tree index. But at 290K QPS this is a DB bottleneck. DB contention with write traffic from index updates.

✅ GREAT — Redis Sorted Set with ZRANGEBYLEX

Store all suggestion strings in Redis Sorted Sets keyed by prefix groups. Score = 0 (alphabetical). ZRANGEBYLEX suggest:{prefix} [prefix] [prefix\xff] returns all suggestions starting with prefix in O(log N + M). Entire dataset: <1GB RAM. Sub-5ms at 290K QPS. Nightly job recomputes from search logs and atomically replaces sorted set. No read/write contention — reads always see a complete, consistent set.

NFR: INDEX FRESHNESS < 30s

How do you reflect price changes in search results within 30 seconds during a flash sale?

❌ BAD — Nightly full re-index

Batch job re-indexes all 500M products every night. Search results show yesterday's prices. During a Black Friday flash sale with 40% price drops, customers see wrong prices. Purchases at stale prices cause order cancellations and legal issues.

⚠ GOOD — Polling the catalog DB every 60s

Polling job SELECT * WHERE updated_at > last_poll. 60s lag, 580 QPS index updates, plus constant scan load on the production DB. At 500M products with frequent price changes, full scans are prohibitive. Misses high-frequency updates within the polling window.

✅ GREAT — CDC via Debezium → Kafka → partial ES update

Debezium reads Postgres WAL → publishes row changes to Kafka topic product.changes. Index Consumer reads events, issues ES _update with only the changed fields (price, in_stock) — no full document re-index. Partial update: <5ms ES write. End-to-end lag: CDC delay (~1s) + Kafka consumer lag (~2s) + ES index refresh interval (1s) = 4–5s total. ES index refresh tunable per index: set to 1s for price/inventory fields, 30s for description (less critical). Also proactively evict Redis query cache entries for affected product categories.

Walmart.com at Scale: Walmart.com handles 250M+ product searches per day across 500M+ SKUs. Their search team (OneSearch platform) runs Elasticsearch with proprietary relevance scoring that combines BM25 with gradient-boosted behavioral signals retrained weekly. During Black Friday, they pre-warm Redis caches for the top-10,000 search queries 2 hours before midnight and set ES index refresh interval to 500ms for price/inventory fields to handle flash sale pricing in near-real-time.
