01 Feature Selection

Payment Authorization & Capture
Process payments end-to-end: initiate, authorize with Visa/MC/PayPal, and capture funds. Maintain an immutable ledger of every state transition.
FEATURE 02
Idempotent Retry & Refund
Handle network retries without double-charging. Process partial and full refunds with at-most-once execution guarantees via idempotency keys.
FEATURE 03

Batch captured payments into end-of-day settlement runs to the bank. Reconcile ledger against bank statements and flag discrepancies.

04 Core Entities
Interview tip: Define entities before API — they anchor your data model and make API shapes obvious. The key question here: what is mutable vs append-only?
💳 PaymentOrder

The top-level intent: a customer wants to pay for an order. Has a lifecycle tracked by replaying ledger entries. Never mutated directly.

payment_order_id · order_id · amount_cents · currency · idempotency_key · created_at
📋 PaymentLedgerEntry

Immutable, append-only record of every state transition. The source of truth. Never UPDATE this table — only INSERT. Current state is derived by replaying entries.

entry_id · payment_order_id · event_type · amount_cents · gateway_ref · created_at
🔐 PaymentToken

Tokenized card reference stored in the PCI vault. No raw PAN ever leaves the vault. Services receive only the token, which is useless if stolen.

token_id · last4 · card_brand · expiry_month · expiry_year · gateway_token · user_id
🏦 GatewayResponse

Raw async response from Visa/Mastercard/PayPal. Stored verbatim for audit. The gateway consumer translates this into a ledger entry.

response_id · payment_order_id · gateway · status_code · auth_code · received_at
💸 Refund

A reversal linked to a captured PaymentLedgerEntry. Partial or full. Has its own idempotency key — a retry must not issue a double refund.

refund_id · original_entry_id · amount_cents · reason · idempotency_key · status
📊 SettlementBatch

End-of-day grouping of CAPTURED payments sent to the bank via wire. One batch per gateway per day. Reconciliation compares batch total vs bank confirmation.

batch_id · settlement_date · gateway · total_amount_cents · status · completed_at
Key relationship: PaymentLedgerEntry is the append-only history; PaymentOrder state is derived by replaying it. Like double-entry bookkeeping — you only add rows, you never erase them.

Idempotency is non-negotiable: Every mutation endpoint requires an idempotency_key. Retrying with the same key returns the cached response without re-executing. A retry on a failed authorization must not issue a second charge.

08 Tradeoffs & Deep Dives
Interview tip: Each deep dive is triggered by an NFR. Use the BAD → GOOD → GREAT escalation pattern. Explain why the naive approach fails before proposing the better one.
NFR: EXACTLY-ONCE CHARGE

How do you prevent double-charging when the checkout service retries on timeout?

❌ BAD — No idempotency

Checkout service retries POST /v1/payments on timeout. Payment Service processes each request independently. Customer charged 2× for one order. Support tickets spike. Chargebacks follow. Fails at production scale.

⚠ GOOD — DB unique constraint on idempotency_key

Add UNIQUE(idempotency_key, event_type) to the ledger table. Duplicate INSERT throws constraint violation → return original response. Works but: DB write on every request at 580 TPS adds contention; doesn't return the original response body cleanly; slow under high retry load.

✅ GREAT — Redis idempotency cache as fast path

Before any processing: GET idem:{idempotency_key} from Redis. Cache hit → return stored JSON response in <1ms, no DB write, no gateway call. Cache miss → process normally, then SET idem:{key} = response EX 86400. DB unique constraint remains as safety net if Redis is cold. Result: zero double-charges, sub-1ms overhead on retries, DB constraint catches the rare Redis miss.

NFR: IMMUTABLE AUDIT TRAIL

How do you maintain payment state without risking data corruption from concurrent updates?

❌ BAD — Mutable status column

Single payments table with a status column. UPDATE payments SET status='AUTHORIZED' WHERE id=X. Race conditions when two concurrent events try to update. Lost history — you can't reconstruct what happened when. Audit logs stored separately get out of sync. Fails PCI audit requirements.

⚠ GOOD — Optimistic locking with version column

UPDATE payments SET status='AUTHORIZED', version=version+1 WHERE id=X AND version=3. Prevents concurrent corruption. But still mutable state — you lose history, can't reconstruct timeline, UPDATE on hot row creates lock contention at 580 TPS.

✅ GREAT — Append-only ledger, state derived from replay

Never UPDATE the ledger. Only INSERT. Current state = SELECT event_type FROM payments_ledger WHERE payment_order_id=X ORDER BY created_at. State machine transitions enforced in application layer: INITIATED→AUTHORIZED is valid, CAPTURED→INITIATED is rejected. Full audit trail is the table itself. Concurrent inserts are fine — each is its own row. Zero lock contention. PCI audit-ready by design.

NFR: ATOMICITY ACROSS SERVICES

How do you ensure inventory deduction + payment + order creation either all succeed or all roll back?

❌ BAD — Two-phase commit (2PC)

Distributed transaction coordinator holds locks across Inventory, Payment, and Order DBs during prepare phase. One slow service blocks all. Gateway timeout (up to 30s) means 30s of locked inventory. At 580 TPS this is a system-wide bottleneck. Not viable.

⚠ GOOD — Outbox pattern

Each service writes to its local DB and an outbox table in the same transaction. CDC reads outbox and publishes to Kafka. At-least-once delivery. Better, but requires compensating transactions to be explicitly coded for each failure scenario. Complex to maintain.

✅ GREAT — Choreography-based Saga via Kafka

Each service listens to Kafka events and publishes its own. Flow: Inventory Service reserves stock → publishes inventory.reserved → Payment Service authorizes → publishes payment.authorized → Order Service confirms → publishes order.confirmed. Compensating events: payment.failed → Inventory Service publishes inventory.reservation.cancelled. No coordinator, no blocking. Each service is autonomous. Failure handling is explicit, testable, and traceable in Kafka topic history.

Walmart at Scale: Walmart processes $500B+ in annual GMV. Their payment system uses WalmartPay tokenization (no raw PAN in any microservice) and a choreography-based Saga for distributed checkout. On Black Friday, their payment authorization service handles 50K+ TPS by keeping the synchronous path (ledger write + idempotency cache check) under 50ms and offloading all gateway communication to Kafka-mediated async workers with independent scaling.
