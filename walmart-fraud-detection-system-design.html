01Feature Selection
Interview tip: Fraud detection has two completely different time horizons — synchronous (<100ms, blocking checkout) and asynchronous (model training, batch analysis). Separate them immediately; mixing them is a common mistake.
FEATURE 01
Real-Time Transaction Scoring
Synchronous accept/reject decision on every transaction in <100ms. Blocking the checkout path — must never add visible latency.
FEATURE 02
Rule Engine for Hard Blocks
Deterministic velocity checks, stolen card blocklist lookups, and geo-anomaly detection. Runs before ML — short-circuits known bad actors in <5ms.
FEATURE 03
Feedback Loop & Model Retraining
Disputed decisions and confirmed fraud feed labeled data back to retrain the ML model. Shadow mode deployment before production promotion.
⬇ Below the Line (Out of Scope)
Account takeover detection  ·  Coupon abuse system  ·  Chargeback dispute automation  ·  Human review queue UI  ·  Biometric authentication
Why mention it: Each is a separate domain. Naming them shows you understand the fraud ecosystem without getting derailed.
02Non-Functional Requirements
Interview tip: The core tension in fraud detection is latency vs accuracy. A more complex model catches more fraud but takes longer. Make this tradeoff explicit per feature.
Real-Time Scoring
p99 < 100ms End-to-End
This is on the checkout critical path. Any latency over 100ms directly harms conversion. Feature store + in-process ML is mandatory.
Real-Time Scoring
False Positive Rate < 0.1%
Blocking a legitimate customer is costly — they don't retry. At 5M txns/day, 0.1% FPR = 5,000 blocked legitimate customers/day.
Rule Engine
< 5ms Rule Evaluation
Rules must be deterministic and hot-reloadable. New rules deployable without service restart (config-driven, not code-driven).
Feedback Loop
Model Retrain < 24h
From labeled feedback to new model in production (via shadow): <24h. Fraudsters adapt daily — model must keep pace.
System-Wide
Full Explainability
Every REJECT must log exactly which rules and features triggered it. Required for dispute resolution and regulatory audit.
System-Wide
99.99% Availability
Fraud service down = checkout blocked or unprotected. Fail-open (approve all) is the safer degradation vs fail-closed (block all).
⬇ Below the Line (Out of Scope)
Real-time chargeback prediction  ·  Integration with Visa/MC fraud networks  ·  GDPR right-to-erasure from feature store  ·  Multi-model ensemble serving

04Core Entities
Interview tip: Fraud detection has a clear split: online entities (Transaction, FraudDecision, FeatureVector) that must be fast, and offline entities (FeedbackEvent, ModelVersion) that can be slow. Name this split explicitly.
💳 Transaction

The event being scored. Carries identity signals (user, device, IP) and transaction context (amount, merchant, type).

transaction_id · type · amount · user_id · device_id · ip · merchant_id · occurred_at
🎯 FraudDecision

The immutable output of every scoring call. Stores the feature snapshot and triggered rules for explainability. Never UPDATE this record.

decision_id · transaction_id · score · action · triggered_rules[] · feature_snapshot · model_version · latency_ms
📊 FeatureVector

Pre-computed risk signals per user/device. Kept fresh by streaming updates on every transaction. The most performance-critical entity.

user_id · txn_velocity_1h · txn_velocity_24h · avg_amount · distinct_merchants · country_mismatch · is_new_device
📋 FraudRule

A deterministic rule in the rule engine. Config-driven so new rules can be deployed without a code release. Evaluated before ML.

rule_id · name · condition_dsl · action (REJECT|REVIEW) · severity · is_active · updated_at
🔄 FeedbackEvent

A labeled ground-truth outcome. Customer dispute resolved → TRUE_POSITIVE or FALSE_POSITIVE. Feeds the training pipeline.

feedback_id · decision_id · outcome (TP|FP|TN|FN) · source (DISPUTE|CHARGEBACK|MANUAL) · occurred_at
🤖 ModelVersion

A trained ML artifact with performance metrics. Can be in SHADOW or PRODUCTION state. Shadow runs in parallel without impacting decisions.

model_id · version · algorithm · auc_roc · false_positive_rate · trained_on · is_shadow · promoted_at
Key relationship: FraudDecision bridges Transaction (what happened) and FeedbackEvent (what was true). Like a court verdict that can be appealed — the Decision is immutable, but FeedbackEvent adds ground truth that improves future verdicts.

08Tradeoffs & Deep Dives
Interview tip: Fraud detection deep dives always circle back to the latency vs accuracy tradeoff. Make it explicit — every architectural choice is a point on that curve.
NFR: SYNCHRONOUS SCORING < 100ms

How do you make an ML inference decision in under 100ms on the checkout critical path?

❌ BAD — Synchronous microservice calls for features + inference

Checkout calls Feature Service (REST, 40ms) → then calls ML Inference Service (REST, 40ms) in serial. 80ms+ just in network overhead before business logic. Any service slowdown pushes past 100ms. Two points of failure on the checkout critical path.

⚠ GOOD — Parallel fan-out to feature service + ML service

Fetch features and run ML inference in parallel goroutines/threads. Reduces to max(40ms, 40ms) = ~40ms. Better, but still 2 network calls. Feature service becomes a bottleneck at 1M RPS. ML service scaling independent from feature scaling.

✅ GREAT — Redis feature store + in-process ONNX model

Three-layer optimization: (1) Feature store in Redis co-located with scoring pods — single HMGET fetches all 20 features in <5ms. (2) ML model loaded in-process as ONNX binary — zero network hop, gradient boosting on 20 features runs in ~10ms. (3) Rule engine (in-memory, ~1ms) runs first — known bad actors short-circuit before any feature fetch. Result: p99 ~50ms with headroom to spare. Only dependency: Redis — no ML service to scale separately.

NFR: FEATURE FRESHNESS FOR VELOCITY FRAUD

How do you detect a fraudster making 10 transactions in 5 minutes if features are precomputed?

❌ BAD — Nightly batch feature computation

txn_velocity_1h computed from yesterday's data. A fraudster executing 10 rapid transactions is invisible until the next batch run. By then, $50K in fraud has cleared. Completely misses the defining signal for velocity fraud.

⚠ GOOD — Query transaction DB at scoring time

SELECT COUNT(*) FROM transactions WHERE user_id=X AND occurred_at > NOW()-1h at scoring time. Fresh data, but adds a DB query to every checkout call. At 50K TPS this is 50K DB queries/second. DB becomes the bottleneck. 20–50ms per query blows the 100ms budget.

✅ GREAT — Dual-write streaming feature updates

Every transaction triggers two async writes: (1) Online: Kafka consumer increments Redis counter INCR velocity:{user_id}:1h and sets EXPIREAT to current_epoch_hour+1. Atomic, <1ms. (2) Offline: nightly Spark job computes heavier 30-day signals and writes to Redis with 24h TTL. Scoring reads online counters (fresh, <1s lag) + offline signals (warm, <24h lag) in a single HMGET. Velocity fraud caught within seconds of the first suspicious transaction.

NFR: MODEL MUST IMPROVE FROM FEEDBACK

How do you safely deploy an improved fraud model without risking a spike in false positives?

❌ BAD — Manual retraining, deploy to production directly

Data scientist retrains model monthly, deploys directly to production. If new model has higher FPR due to data distribution shift, thousands of legitimate customers blocked before anyone notices. Rollback is manual and slow.

⚠ GOOD — A/B testing with traffic split

Route 5% of traffic to new model, 95% to champion. Compare metrics. Problem: at 5% of 50K TPS = 2,500 TPS, real users get blocked by the challenger model during the test window. Acceptable for low-stakes features, not for checkout blocking decisions.

✅ GREAT — Shadow mode with automated promotion gate

New model runs in SHADOW: scores every transaction in parallel with production model, logs shadow_score to Kafka. Shadow decisions never affect the customer. After 48h shadow window: compare shadow AUC-ROC vs champion. Auto-promote gate: shadow AUC > champion + 0.005 AND shadow FPR < current FPR AND zero incidents in 48h → auto-promote via feature flag. Rollback gate: if production FPR spikes 2× after promotion → auto-revert to previous champion in <60s. Zero customer impact during evaluation; fast safe rollback if anything goes wrong.

Walmart at Scale: Walmart loses ~$3B annually to retail fraud including return fraud, payment fraud, and coupon abuse. Their fraud platform runs 500+ active rules updated daily by the fraud ops team, combined with a gradient boosting model trained on 5M+ labeled transactions weekly. On Black Friday, they pre-load the stolen card bloom filter with 48h of proactive threat intelligence from Visa/Mastercard networks, blocking known compromised cards before they attempt checkout. Their shadow deployment system allows weekly model refreshes with a 48-hour safety window before any customer impact.
